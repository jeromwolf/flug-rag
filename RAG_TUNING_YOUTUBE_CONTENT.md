## 1. 우리가 만든 시스템

### 기술 스택

```
검색:   Vector(bge-m3, 1024dim) + BM25(kiwipiepy 형태소) → CrossEncoder 리랭킹
생성:   Qwen 2.5 (7B → 14B) via Ollama
저장:   ChromaDB
평가:   Custom Evaluator (Semantic + ROUGE + Length Penalty)
```

### 대상 문서

| 데이터셋 | 문서 수 | 유형 | 골든 문항 수 |
|---------|--------|------|------------|
| 법률 문서 | PDF + 관련법 | 법 조문, 시행령 | 50 |
| 내부 규정 | 676 HWP | 인사, 감사, 보수, 복무 등 | 60 |

| 홍보 자료 | 7 PDF | 브로셔, 회사사 | 20 |
| 공시 자료 | 403 files | 재무제표, 감사보고서 | 20 |

**총 120문항, 4개 카테고리**: factual(사실확인), inference(추론), multi_hop(다중규정연계), negative(부재확인)

## 2. 60%에서 시작하다 — 현실은 냉혹했다

첫 번째 벤치마크를 돌렸을 때 성공률 **60%**. 기대했던 것과는 전혀 달랐습니다.

### 처음에 발생한 주요 문제들

```
❌ "한국가스공사법에 대해 질문했는데, 내부규정을 인용해서 답변"
❌ "종류를 물었더니 각 항목의 설명까지 장황하게 붙임"
❌ "규정이 없는데 관련 단어가 있다는 이유로 있다고 답변"
❌ "중국어가 갑자기 튀어나옴" (Qwen 모델 특성)
❌ "답변이 300자가 넘어가면서 핵심을 놓침"
```

### 깨달은 것

> **RAG 시스템에서 검색(Retrieval)은 50%만 해결한다.**
> **나머지 50%는 프롬프트 엔지니어링 + 평가 체계 + 후처리다.**

처음에는 "검색만 잘 되면 되겠지" 생각했습니다. 하이브리드 검색(Vector + BM25)에 리랭커까지 붙였으니까요. 하지만 검색이 정확해도 LLM이 엉뚱하게 답변하면 소용없습니다.

## 3. 골든 데이터셋: 평가 없이 튜닝 없다

### 왜 골든 데이터셋이 먼저인가

튜닝을 시작하기 전에 **"뭘 기준으로 좋고 나쁨을 판단할 것인가"**가 먼저입니다.
감으로 "이 정도면 괜찮겠지"는 통하지 않습니다.

### 골든 데이터셋 구조

```json
{
    "id": 1,
    "category": "factual",
    "question": "감사규정의 목적은 무엇인가요?",
    "answer": "감사 직무수행의 기준과 절차를 정함으로써...",
    "source_regulation": "감사규정",
    "difficulty": "easy"
}
```

각 문항에 반드시 포함해야 할 것:
- **category**: 질문의 유형 (후에 카테고리별 가중치 평가에 사용)
- **answer**: 기대 정답 (평가 기준선)
- **difficulty**: easy/medium/hard (실패 분석 시 난이도별 패턴 파악)
- **source**: 출처 문서 (검색 정확도 검증용)

### 4가지 카테고리 설계 이유

| 카테고리 | 비율 | 왜 필요한가 | 예시 |
|---------|------|-----------|------|
| **factual** (사실확인) | 40% | 기본기. 이게 안 되면 아무것도 안 됨 | "징계의 종류는?" |
| **inference** (추론) | 30% | 규정을 읽고 "왜 그런지" 설명하는 능력 | "과목당 4할 기준이 있는 이유는?" |
| **multi_hop** (다중연계) | 20% | 여러 문서를 연결해야 답할 수 있는 질문 | "채용과 인사규정의 관계는?" |
| **negative** (부재확인) | 10% | "없다"고 정확히 답할 수 있는 능력 | "퇴직금 산정 규정이 있나요?" |

> **핵심 인사이트**: negative 카테고리는 비율은 10%지만 가장 튜닝하기 어려웠습니다.
> LLM은 본능적으로 "뭔가 답을 만들어내려" 합니다. "없다"고 말하게 만드는 게 가장 어렵습니다.

### 골든 데이터셋 만드는 3가지 방법

#### 방법 1: 수동 생성 (가장 정확, 가장 느림)

전문가가 직접 문서를 읽고 Q&A를 작성합니다.

```
장점: 정확도 최상, 도메인 뉘앙스 반영
단점: 시간이 많이 걸림 (문항당 10-20분)
추천: 첫 50문항은 수동으로 만들 것
```

#### 방법 2: LLM 기반 자동 생성 + 검증

우리가 만든 자동 생성기의 작동 방식:

```
Step 1. 청크 후보 선별 (채점 휴리스틱)
  - 숫자가 포함된 청크 → factual 후보 (+2.0점)
  - "제N조" 패턴 → factual 후보 (+3.0점)
  - "하여야 한다" 패턴 → inference 후보 (+2.0점)
  - "경우에는" 패턴 → inference 후보 (+1.5점)

Step 2. 카테고리별 프롬프트로 Q&A 생성

Step 3. 자기 일관성 검증 (Self-Consistency)
  - 같은 소스 청크로 답변을 다시 생성 (temperature 0.1)
  - 원래 답변과 비교 (유사도 0.6 이상이어야 통과)
  - 일관성 없으면 폐기

Step 4. 중복 제거 (Jaccard 유사도 > 0.8이면 폐기)

Step 5. 난이도 자동 판정
  - 단일 청크 + 짧은 답변 = easy
  - 다중 청크 또는 긴 답변 = hard
  - 추론 마커 포함 = medium
```

**자동 생성기 CLI 예시:**

```bash
python tests/golden_generator.py \
  -n "내부규정" \
  --filename-pattern ".*규정.*" \
  --counts '{"factual":8,"inference":6,"multi_hop":4,"negative":2}'
```

#### 방법 3: 하이브리드 (추천)

```
1단계: LLM으로 100문항 자동 생성
2단계: 전문가가 걸러내기 (약 30-40% 탈락)
3단계: 탈락 문항 대체 수동 생성
4단계: 최종 데이터셋 검증 실행
```

### 데이터셋 검증 체크리스트

```
✅ 카테고리 균형: factual 40%, inference 30%, multi_hop 20%, negative 10%
✅ 난이도 분포: easy 30-40%, medium 30-40%, hard 20-30%
✅ 중복 검사: Jaccard 유사도 0.8 이상 쌍 없음
✅ 답변 길이: 50-200자 범위
✅ 출처 확인: 모든 문항에 source 명시
✅ 문항 수: 데이터셋당 최소 20문항 (통계적 의미)
```

## 4. 평가 시스템 설계: 한국어 RAG는 다르다

### 왜 기존 평가 방식이 안 통하는가

영어 RAG 평가에서 흔히 쓰는 방법:
- BLEU Score → 한국어에 부적합 (어순 자유)
- Word-level ROUGE → 한국어에 부적합 (교착어 — 조사가 붙으면 다른 단어)
- Exact Match → 한국어에 부적합 (같은 의미, 다른 표현이 너무 많음)

### 우리의 평가 공식

#### 기본 Composite Score

```
composite = (0.65 × semantic + 0.20 × rouge_l + 0.15 × rouge_1) × length_penalty
```

**왜 이 비율인가:**
- 한국어 법률 QA에서 같은 법 개념을 다양하게 표현할 수 있음
- "법인" = "법인격을 가진 단체" = "법적 인격체" → 의미는 같지만 글자가 다름
- 따라서 의미 유사도(semantic) 비중을 65%로 높게 잡음
- 하지만 법 조문 인용이 필요한 경우 원문 일치도(ROUGE)도 필요

#### 카테고리별 가중치 차별화

```python
CATEGORY_WEIGHTS = {
    "factual":   {"semantic": 0.55, "rouge_l": 0.30, "rouge_1": 0.15},
    "inference":  {"semantic": 0.70, "rouge_l": 0.15, "rouge_1": 0.15},
    "multi_hop":  {"semantic": 0.65, "rouge_l": 0.20, "rouge_1": 0.15},
    "negative":   {"semantic": 0.80, "rouge_l": 0.10, "rouge_1": 0.10},
}
```

**설계 근거:**
- **factual**: 법 조문 원문 인용이 중요 → ROUGE 비중 높임 (0.30)
- **inference**: 해석의 자유도가 높음 → semantic 비중 높임 (0.70)
- **negative**: "없다"의 표현이 매우 다양 → semantic 거의 전부 (0.80)
- **multi_hop**: 여러 문서 연결이라 균형 필요 → 중간 (0.65)

#### 한국어 ROUGE: 글자 단위로 해야 하는 이유

```python
# 일반적인 word-level tokenizer
"감사규정의 목적은" → ["감사규정의", "목적은"]  # "의", "은" 조사 때문에 매칭 실패

# 우리의 character-level tokenizer
"감사규정의 목적은" → ["감", "사", "규", "정", "의", "목", "적", "은"]
```

한국어는 교착어입니다. 조사(은/는/이/가/의/를)가 명사에 붙기 때문에 단어 단위로 비교하면 같은 내용도 다르게 판정됩니다. **글자 단위 ROUGE**로 바꾸니 평가 정확도가 확 올라갔습니다.

추가로 ROUGE 계산 전에 이런 노이즈를 제거합니다:

```python
text = re.sub(r"\[출처[^\]]*\]", "", text)    # [출처: ...] 태그 제거
text = re.sub(r"^[AQ][:：]\s*", "", text)      # A:/Q: 접두사 제거
```

#### 길이 패널티: 장황한 답변 잡기

```
답변 길이가 기대 길이의 2.5배 이하 → 패널티 없음 (1.0)
2.5배 초과 시 → 점수 차감 시작
최대 30% 감점 (penalty = 0.7)
```

**공식:**

```python
if actual_length <= expected_length * 2.5:
    penalty = 1.0
else:
    ratio = actual_length / expected_length
    penalty = max(0.7, 1.0 - (ratio - 2.5) * 0.04)
```

**왜 필요한가:** "징계의 종류는?"이라고 물었을 때:
- 좋은 답변: "파면, 해임, 강등, 정직, 감봉, 견책입니다." (30자)
- 나쁜 답변: "징계의 종류는 다음과 같습니다. 1) 파면: 공무원 관계를 소멸시키는... 2) 해임: 직위를 박탈하는... (후략)" (300자)

의미 유사도만으로는 둘 다 높은 점수를 받습니다. 길이 패널티가 후자를 잡아줍니다.

#### 부정 질문 감지 (Negation-Aware Scoring)

```python
negation_markers = [
    # 내부규정 특화 (14개)
    "확인되지 않습니다", "규정은 없습니다", "규정이 없습니다",
    "포함하고 있지 않습니다", "명시되어 있지 않습니다",
    "별도로 규정하고 있지", "직접적인 규정은 없",
    "해당 규정이 없습니다", "규정하고 있지 않",
    "관한 규정은 없", "관련 규정은 없",
    "규정은 포함하고 있지", "절차는 명시되어 있지",
    "해당 내용은 확인되지",

    # 일반 도메인 확장 (17개)
    "제공하지 않습니다", "포함되어 있지 않습니다",
    "확인할 수 없습니다", "정보는 없습니다",
    "내용은 없습니다", "다루고 있지 않",
    "기재되어 있지 않", "수록되어 있지 않",
    "찾을 수 없", "없는 것으로",
    # ... 총 30+개
]

# 정규식 폴백 (새로운 표현 자동 감지)
neg_pattern = re.compile(
    r'(확인되지|포함되어 있지|포함하고 있지|제공하지'
    r'|없습니다|않습니다|않았다|없다고|없으며|없음)'
)
```

기대 답변이 "없다"인데 실제 답변이 "있다"면 → **점수 × 0.3 (70% 감점)**

30+개 패턴 + 정규식 폴백으로 다양한 도메인의 부정 표현을 커버합니다. 이 기능 하나로 negative 카테고리 정확도가 크게 올라갔습니다.

#### 등급 체계

| 등급 | 기준 | 의미 |
|------|------|------|
| A | >= 0.80 | 우수 — 답변 품질 높음 |
| B | >= 0.65 | 양호 — 실용적으로 충분 |
| C | >= 0.50 | 보통 — 개선 필요하지만 사용 가능 |
| D | >= 0.35 | 미흡 — 부정확하거나 불완전 |
| F | < 0.35 | 실패 — 완전히 잘못된 답변 |

## 5. Phase 1: 60% → 100% — 프롬프트의 힘

### 놀라운 사실: 파라미터를 건드리지 않았다

Phase 1에서 60%를 100%로 올리는 데 **RAG 파라미터는 하나도 바꾸지 않았습니다.**

```
top_k = 20       (기본값 유지)
rerank_top_n = 5  (기본값 유지)
temperature = 0.3 (기본값 유지)
vector_weight = 0.7 (기본값 유지)
bm25_weight = 0.3   (기본값 유지)
```

**변경한 것은 오직 프롬프트뿐.**

### 6가지 프롬프트 개선 기법

#### 기법 1: 출처 분리 지시

**문제**: 법률 문서와 내부 규정이 섞여서 검색되면 LLM이 잘못된 문서를 인용

**해결:**
```
[중요: 문서 출처 확인]
- 각 문서의 출처(파일명)를 반드시 확인하세요.
- 질문이 "A법"에 대한 것이면, "B사" 내부규정은 무시하세요.
- 질문이 내부규정에 대한 것이면, 해당 규정 문서만 참조하세요.
```

**효과**: 이것 하나로 교차 오염(cross-contamination) 문제 해결. **약 15% 성공률 개선.**

#### 기법 2: 간결성 강제 규칙

**문제**: 7B 모델이 질문의 2-3배 길이로 장황하게 답변

**해결:**
```
4. 1~3문장으로 간결하게 답변하세요.
4-1. 답변은 50~150자를 목표로 하세요. 200자를 초과하지 마세요.
5. 목록형 질문("종류는?", "구성은?")에는 항목 이름만 나열하세요.
   각 항목의 기간, 금액, 내용, 설명을 절대 덧붙이지 마세요.
```

**핵심 발견 — 모델 크기별로 간결성 지시 강도를 다르게 해야 합니다:**

```
7B 모델:  "반드시 1~3문장으로만 답변하세요. 추가 해석을 절대 덧붙이지 마세요."
14B 모델: "50~150자로 작성하세요. 200자를 절대 초과하지 마세요."
70B+ 모델: (추가 지시 불필요)
```

작은 모델일수록 더 강력한 제약이 필요합니다. 큰 모델은 기본 지시만으로 충분합니다.

#### 기법 3: 원문 인용 강제

**문제**: LLM이 법 조문을 의역해서 정확도 하락

**해결:**
```
2. 찾은 조문의 원문을 최대한 그대로 인용하세요. 원문의 표현을 바꾸거나 의역하지 마세요.
```

ROUGE 점수가 급상승. 특히 factual 카테고리에서 **ROUGE-L이 0.3 → 0.5**로 개선.

#### 기법 4: 도메인 특화 Few-Shot

일반적인 few-shot이 아니라, **실패 패턴별로 맞춤 예시**를 추가했습니다:

```yaml
# 패턴 1: 단순 사실 (조문 인용)
Q: "법적 지위는 무엇인가요?"
A: "법인입니다. (제2조)"

# 패턴 2: 숫자 사실 (정확한 수치)
Q: "소멸시효는 얼마인가요?"
A: "원금은 5년, 이자는 2년입니다. (제14조 제4항)"

# 패턴 3: 목록형 (이름만 나열)
Q: "징계의 종류는?"
A: "파면, 해임, 강등, 정직, 감봉, 견책입니다."

# 패턴 4: 부정 (없다고 답하기)
Q: "퇴직금 산정 방법을 규정하고 있나요?"
A: "퇴직금 산정 방법에 대한 규정은 확인되지 않습니다."

# 패턴 5: 추론 (이유 설명)
Q: "과목당 4할 기준이 있는 이유는?"
A: "특정 분야에 역량 부족이 있는 지원자가 다른 과목의 높은 점수로 만회하여 합격하는 것을 방지하기 위함입니다."
```

**총 30개의 few-shot 예시를 도메인별(법률 20, 일반 8, 기술 2)로 분류해서 적용.**

질문이 들어오면 검색된 문서의 메타데이터로 도메인을 자동 판별하고, 해당 도메인의 few-shot만 프롬프트에 포함합니다.

#### 기법 5: 부정 질문 전용 규칙

**이것이 가장 어려웠습니다.**

**문제**: "퇴직금 규정이 있나요?"라고 물으면, 보수규정에 "퇴직금"이라는 단어가 한 번 언급되어 있다는 이유로 "있습니다"라고 답변

**해결 — 3번의 반복 끝에 완성된 규칙:**
```
[부정 질문 처리 - 최우선 엄격 준수]
1. 해당 주제를 전담으로 다루는 독립적 조항이 있는지 판단하세요.
2. 단순히 관련 단어가 다른 맥락에서 언급되는 것은 "규정이 있다"가 아닙니다.
   예: "퇴직금"이 보수규정에 단어로 등장해도, 퇴직금 산정 방법을 전담 규정하는
       조항이 없으면 "확인되지 않습니다"
3. 확신이 없으면 반드시 "확인되지 않습니다"로 답하세요.
   거짓 긍정(False Positive)보다 거짓 부정(False Negative)이 낫습니다.
```

**핵심 인사이트:**
> "거짓 긍정보다 거짓 부정이 낫다"
> LLM에게 이 원칙을 명시적으로 알려줘야 합니다. 그렇지 않으면 LLM은 항상 뭔가를 만들어냅니다.

#### 기법 6: 금지 규칙 명시

```
[절대 금지]
- 중국어, 영어, 일본어, 한자 사용 금지
- 컨텍스트에 없는 조문번호나 법률 내용을 만들어내지 마세요
- "[출처: ...]" 태그를 붙이지 마세요
- 면책 문구("정확하지 않을 수 있습니다") 금지
```

Qwen 계열 모델은 가끔 중국어가 섞여 나옵니다. 명시적으로 금지하고, 후처리에서도 한자를 제거합니다:

```python
text = re.sub(r'[\u4E00-\u9FFF\u3040-\u309F\u30A0-\u30FF]+[^\n]*$', '', text)
```

### Phase 1 결과

```
Before: 60% (30/50)
After:  100% (50/50)
평균 신뢰도: 0.999

변경한 것: 프롬프트만
변경하지 않은 것: RAG 파라미터 전부
```

"감사규정에서..."으로 시작하는 질문이 오면 자동으로 `source_type: "내부규정"`을 필터에 추가.

```
2. 그 규정이 존재하는 실질적 이유, 방지하려는 문제를 설명하세요.
```

### 회귀(Regression)에서 배운 것

Phase A에서 `context_max_chunks=4`와 `llm_max_tokens=1024`를 동시에 적용했습니다. max_tokens 줄이기는 장황한 답변을 줄여 좋은 효과가 있었지만, context_max_chunks=4가 치명적이었습니다.

**왜 context_max_chunks=4가 위험했나:**
- 리랭커가 5개 청크를 선별한 후 다시 4개로 잘라냄
- 부정 질문에서 "관련 없는 내용이 없다"는 판단에 필요한 맥락이 사라짐
- negative 카테고리가 ~100%에서 45.5%로 붕괴

> **교훈: 두 가지 변경을 동시에 적용하면 어떤 것이 개선이고 어떤 것이 악화인지 구분할 수 없다. 항상 하나씩 변경하고 측정하라.**

## 8. Phase 3 심화: 회귀 진단과 복구

### 회귀 탐지 프로토콜

파라미터 변경 후 벤치마크를 돌렸을 때 성능이 떨어지면, 공황에 빠지기 쉽습니다. 체계적인 진단 프로토콜이 필요합니다.

#### Step 1: 변경 격리 (Isolation)

```
문제: context_max_chunks=4 + llm_max_tokens=1024 동시 적용 후 90.8%로 하락
접근: 하나만 롤백하고 다시 측정
결과: context_max_chunks=0으로 복원 → 94.2% 회복
결론: context_max_chunks=4가 범인, max_tokens=1024는 유지 OK
```

#### Step 2: 카테고리별 영향 분석

```
변경 전 → 변경 후 비교:
  factual:    45/48 → 43/48   (-2) — 약간 악화
  inference:  35/36 → 35/36   (±0) — 변화 없음
  multi_hop:  25/25 → 25/25   (±0) — 변화 없음
  negative:   9/11  →  5/11   (-4) — 대폭 악화 ← 범인!
```

특정 카테고리만 크게 영향받았다면, 그 카테고리에 특화된 메커니즘을 의심합니다.

#### Step 3: 근본 원인 분석 (Root Cause)

negative 카테고리 실패 문항을 하나씩 살펴봤더니:

```
Q19 (brochure): cat_score 0.19 → 평가기가 0.3x 패널티 부여
Q19 (travel):   cat_score 0.21 → 평가기가 0.3x 패널티 부여
Q19 (ALIO):     cat_score 0.24 → 평가기가 0.3x 패널티 부여
```

모두 **평가기의 neg_markers가 실제 답변의 부정 표현을 인식하지 못해** 잘못된 패널티를 부여한 것이었습니다!

#### 문제: neg_markers가 너무 좁았다

원래 14개 패턴:

### Temperature 최적화: 0.1 vs 0.2 vs 0.3

Temperature는 단순히 "낮을수록 좋다"가 아닙니다. 데이터셋별로 최적값이 다릅니다.

```
temperature=0.1:
  ✅ negative: 81.8% (9/11) — 최고
  ✅ 홍보자료: 100% (20/20)
  ❌ 출장보고: 70% (14/20) — 대폭 하락!

  이유: 출장보고서는 사람 이름, 지명, 장비명 등 고유명사가 많음.
  temperature가 너무 낮으면 모델이 이름을 "안전한" 일반적 표현으로 대체.

temperature=0.3:
  ✅ 출장보고: 90% (18/20) — 좋음
  ❌ negative: ~60% — 모델이 "뭔가 답변"하려는 경향 강화

  이유: temperature가 높으면 모델의 창의성이 올라가지만,
  부정 질문에서 "없다" 대신 "관련 내용을 찾아보면..." 식으로 답변.

temperature=0.2:  ← 최종 선택
  ✅ negative: 81.8% (9/11) — 양호
  ✅ 출장보고: 80% (16/20) — 수용 가능
  ✅ 홍보/ALIO: 100%

  타협점. 완벽하지 않지만 모든 데이터셋에서 수용 가능한 성능.
```

> **교훈: Temperature는 전체 최적이 아닌, 최악 카테고리를 최소화하는 값을 선택하라.**

### 부정 질문 프롬프트: "현재 vs 과거" 구별

기존 부정 질문 규칙은 "규정"에만 집중했습니다. 하지만 홍보물/출장보고서에서는 다른 패턴의 거짓 긍정이 발생합니다.

**문제 예시:**
```

올바른 답변: "현재 임직원 수는 확인되지 않습니다. 과거 특정 시점의 인원만 언급됩니다."
```

**추가한 규칙:**
```
[부정 질문 처리 - 최우선 엄격 준수]
3. "현재"를 묻는 질문에 과거 데이터만 있으면 "확인되지 않습니다"로 답하세요.
4. 특히 "산정 방법", "관리 체계", "수당 체계" 등 구체적 세부 사항을 묻는 질문에는,
   해당 세부 사항을 명시적으로 다루는 조항이 있어야만 "있다"입니다.
```

### 14B 간결성 지시 분화: factual vs negative

7B 모델에는 "1~3문장으로만"이라는 일률적 지시가 효과적이었습니다. 하지만 14B 모델에서는 질문 유형별로 적정 답변 길이가 다릅니다.

**기존 (일률적):**
```
"50~150자로 작성하세요. 200자를 절대 초과하지 마세요.
 면책 문구를 사용하지 마세요."
```

**개선 (유형별 분화):**
```
"- 사실(factual) 질문: 50~150자로 간결하게 답변하세요.
 - 부정(negative) 질문: 100~250자로, '확인되지 않습니다'와 함께 이유를 설명하세요.
 - 핵심 사실만 답변하고, 컨텍스트의 원문 표현을 그대로 사용하세요.
 - 불필요한 서론, 반복, 부연 설명을 삼가세요."
```

**왜 분화가 필요한가:**
- factual: 조문 원문 인용이면 50자도 충분
- negative: "없습니다" + "왜 없는지" + "대신 무엇이 있는지" 설명 필요 → 최소 100자
- "면책 문구 금지"를 제거: negative 답변에서 "~에 대한 규정은 확인되지 않습니다"는 면책이 아닌 정확한 답변

## 9. 튜닝 알고리즘 완전 가이드

이 섹션은 RAG 시스템 튜닝에 사용한 구체적인 알고리즘과 방법론을 코드 수준에서 설명합니다.

### 알고리즘 1: 카테고리별 가중치 평가 (Category-Weighted Scoring)

기본 composite score는 모든 질문에 동일한 가중치를 적용합니다. 하지만 질문 유형별로 "정확한 답변"의 기준이 다릅니다.

#### 핵심 아이디어

```
factual 질문  → 법 조문을 정확히 인용했는가? → ROUGE 비중 ↑
inference 질문 → 의미를 제대로 해석했는가? → Semantic 비중 ↑
negative 질문  → "없다"를 정확히 판별했는가? → Semantic 비중 ↑↑
```

#### 알고리즘 상세

```python
def compute_category_score(eval_result, category, expected, actual):
    """카테고리별 가중치 적용 → 부정 감지 패널티 → 최종 점수"""

    # Step 1: 카테고리별 가중치 선택
    weights = {
        "factual":   {"semantic": 0.55, "rouge_l": 0.30, "rouge_1": 0.15},
        "inference":  {"semantic": 0.70, "rouge_l": 0.15, "rouge_1": 0.15},
        "multi_hop":  {"semantic": 0.65, "rouge_l": 0.20, "rouge_1": 0.15},
        "negative":   {"semantic": 0.80, "rouge_l": 0.10, "rouge_1": 0.10},
    }.get(category, {"semantic": 0.65, "rouge_l": 0.20, "rouge_1": 0.15})

    # Step 2: 가중 합산
    score = (
        weights["semantic"] * eval_result.semantic_similarity
        + weights["rouge_l"] * eval_result.rouge_l
        + weights["rouge_1"] * eval_result.rouge_1
    ) * eval_result.length_penalty

    # Step 3: 부정 감지 패널티 (negative 카테고리만)
    if category == "negative":
        expected_negative = is_negative_answer(expected)  # 골든 답변이 "없다"인지
        actual_negative = is_negative_answer(actual)      # 생성 답변이 "없다"인지

        if expected_negative and not actual_negative:
            score *= 0.3  # 70% 감점!

    return score
```

**설계 근거:**
- factual의 ROUGE 비중이 0.30인 이유: "제14조 제4항"을 정확히 인용해야 함
- negative의 semantic 비중이 0.80인 이유: "없다"를 다양하게 표현할 수 있어서 어휘 일치보다 의미 일치가 중요
- 0.3x 패널티의 근거: 거짓 긍정은 사용자에게 잘못된 정보를 제공하므로 치명적

### 알고리즘 2: 부정 답변 감지 (Negation Detection)

LLM이 "없다"고 정확히 답했는지 판별하는 것은 생각보다 어렵습니다.

#### 왜 어려운가

```
"확인되지 않습니다"          → 명확한 부정 ✅
"제공하지 않습니다"          → 부정이지만 다른 표현 ✅
"관련 내용이 포함되어 있지 않습니다" → 긴 부정 표현 ✅
"해당 규정은 없는 것으로 보입니다"  → 우회적 부정 ✅
"직접적인 규정은 없으나..."    → 부정 + 추가 정보 ✅ (여전히 부정)
```

#### 3단계 감지 알고리즘

```python
def is_negative_answer(text):
    """3단계 부정 답변 감지"""

    # Stage 1: 정확 매칭 (30+개 패턴)
    exact_markers = [
        "확인되지 않습니다", "규정은 없습니다", "규정이 없습니다",
        "포함하고 있지 않습니다", "제공하지 않습니다",
        "포함되어 있지 않습니다", "확인할 수 없습니다",
        "정보는 없습니다", "내용은 없습니다",
        "다루고 있지 않", "기재되어 있지 않",
        "수록되어 있지 않", "찾을 수 없",
        # ... 총 30+개
    ]
    if any(marker in text for marker in exact_markers):
        return True

    # Stage 2: 정규식 폴백 (새로운 표현 커버)
    neg_pattern = re.compile(
        r'(확인되지|포함되어 있지|포함하고 있지|제공하지'
        r'|없습니다|않습니다|않았다|않았습니다'
        r'|없다고|없으며|없음)'
    )
    if neg_pattern.search(text):
        return True

    # Stage 3: 어떤 패턴에도 안 걸리면 긍정으로 판단
    return False
```

**핵심 인사이트:**
- Stage 1만으로는 70% 감지율 (홍보물/출장 도메인 누락)
- Stage 2 추가로 95%+ 감지율
- 새로운 도메인 추가 시 Stage 1에 도메인 특화 패턴 추가 필요

### 알고리즘 3: Temperature 격자 탐색 (Grid Search)

단일 최적값이 아닌, 카테고리별 Trade-off를 분석합니다.

```
Grid: temperature ∈ {0.05, 0.1, 0.15, 0.2, 0.25, 0.3, 0.4, 0.5}

각 temperature에서:
  1. 전체 120문항 벤치마크 실행
  2. 카테고리별 성공률 기록
  3. "최악 카테고리 성공률" 기록

선택 기준: max(min(category_success_rates))
= 최악 카테고리의 성공률을 최대화하는 temperature
```

**실제 결과:**

```
temp  | factual | inference | multi_hop | negative | min  | 선택
------+---------+-----------+-----------+----------+------+-----
0.1   | 91.7%   | 97.2%     | 100%      | 81.8%    | 81.8%|
0.2   | 93.8%   | 97.2%     | 100%      | 81.8%    | 81.8%| ← 최종
0.3   | 95.8%   | 100%      | 92.0%     | ~60%     | ~60% |
```

temperature=0.2가 선택된 이유:
- min(카테고리 성공률) = 81.8%로 0.1과 동일하지만
- factual이 93.8% vs 91.7%로 더 높음
- 출장보고서 성능도 80% vs 70%로 더 안정적

### 알고리즘 4: 길이 패널티 (Length Penalty)

장황한 답변을 감점하는 비선형 패널티 함수입니다.

```python
def length_penalty(actual_length, expected_length):
    """
    비선형 길이 패널티 함수

    ratio <= 2.5: penalty = 1.0 (감점 없음)
    ratio = 5.0:  penalty = 0.9 (10% 감점)
    ratio = 10.0: penalty = 0.7 (30% 감점, 최대)
    """
    if expected_length == 0:
        return 1.0

    ratio = actual_length / expected_length

    if ratio <= 2.5:
        return 1.0  # 2.5배까지는 허용

    # 선형 감소: 2.5x에서 시작, 0.04/배수 감점
    penalty = 1.0 - (ratio - 2.5) * 0.04
    return max(0.7, penalty)  # 최소 0.7 (최대 30% 감점)
```

**시각화:**
```
길이 비율:  1.0x  2.0x  2.5x  3.0x  5.0x  7.5x  10.0x
패널티:     1.0   1.0   1.0   0.98  0.90  0.80   0.70
감점률:      0%    0%    0%    2%   10%   20%    30%
```

**왜 2.5배부터인가:** 한국어 답변은 조사 차이로 같은 내용도 길이가 20-50% 달라질 수 있음. 2.5배까지는 이 변동을 흡수하고, 그 이상은 확실히 장황한 것.

### 알고리즘 5: 모델 크기 인식 프롬프팅 (Model-Size-Aware Prompting)

모델 이름에서 파라미터 수를 추출하고, 크기에 맞는 간결성 지시를 추가합니다.

```python
def get_conciseness_suffix(model_hint):
    """모델 크기별 간결성 지시 생성"""

    model_lower = model_hint.lower()

    # 소형 모델 (7B, 8B, 3B 등)
    small = ["7b", "8b", "3b", "1b", "1.5b", "mini", "tiny", "small"]
    if any(p in model_lower for p in small):
        return """
[추가 지시 - 반드시 따르세요]
- 반드시 1~3문장으로만 답변하세요.
- 법 조문의 원문을 그대로 인용하세요.
- 추가 해석이나 부연 설명을 절대 덧붙이지 마세요."""

    # 중형 모델 (14B, 13B 등)
    medium = ["14b", "13b", "12b", "15b", "20b"]
    if any(p in model_lower for p in medium):
        return """
[추가 지시 - 반드시 따르세요]
- 사실(factual) 질문: 50~150자로 간결하게 답변하세요.
- 부정(negative) 질문: 100~250자로, '확인되지 않습니다'와 함께 이유를 설명하세요.
- 핵심 사실만 답변하고, 컨텍스트의 원문 표현을 그대로 사용하세요.
- 불필요한 서론, 반복, 부연 설명을 삼가세요."""

    # 대형 모델 (70B+): 추가 지시 불필요
    return ""
```

**왜 모델 크기별로 다른가:**
- **7B**: 지시를 잘 안 따름 → "반드시", "절대" 같은 강한 표현 필요
- **14B**: 지시를 대체로 따르지만 과잉 적용 위험 → factual/negative 분리 필요
- **70B+**: 기본 시스템 프롬프트만으로 충분 → 추가 지시가 오히려 방해

### 알고리즘 6: 도메인 자동 감지 (Auto Domain Detection)

검색된 문서의 메타데이터를 분석해 질문의 도메인을 자동으로 판별하고, 도메인에 맞는 시스템 프롬프트와 few-shot을 선택합니다.

```python
def detect_document_type(context_chunks):
    """검색된 청크의 메타데이터로 도메인 판별"""

    legal_score = 0
    technical_score = 0
    general_score = 0

    for chunk in context_chunks:
        metadata = chunk["metadata"]
        filename = metadata.get("filename", "").lower()
        source_type = metadata.get("source_type", "")
        content = chunk["content"]

        # 일반 (홍보물, 출장보고서, ALIO)
        if source_type in ("홍보물", "출장보고서", "ALIO공시"):
            general_score += 3

        # 법률/규정
        if source_type in ("법률", "내부규정", "정관"):
            legal_score += 3
        if re.search(r'제\d+조', content):
            legal_score += 1

        # 기술
        if re.search(r'(매뉴얼|지침서|절차서)', filename):
            technical_score += 2

    if legal_score > technical_score and legal_score > general_score:
        return "legal"
    elif technical_score > legal_score and technical_score > general_score:
        return "technical"
    return "general"
```

도메인별로 다른 것:

```
legal    → rag_legal_system 프롬프트 + legal_examples (20개)
technical → rag_technical_system 프롬프트 + technical_examples (2개)
general  → rag_system 프롬프트 + general_examples (12개)
```

### 알고리즘 7: 실패 패턴 자동 분류 (Failure Pattern Classifier)

벤치마크 실패 문항을 10가지 패턴으로 자동 분류하고, 각 패턴에 맞는 수정 제안을 생성합니다.

```python
def classify_failure(question, eval_result, category, confidence):
    """실패 문항을 패턴으로 분류"""

    # 우선순위 순서대로 체크
    if confidence < 0.3:
        return "WRONG_RETRIEVAL"  # 검색 자체가 실패

    if eval_result.length_penalty < 0.9:
        return "VERBOSE_ANSWER"  # 답변이 너무 김

    if eval_result.semantic_similarity < 0.4:
        return "HALLUCINATION"  # 완전히 다른 내용

    if category == "negative" and eval_result.semantic_similarity > 0.4:
        return "WRONG_SCOPE"  # "없다"를 "있다"로 답함

    if category == "inference" and eval_result.rouge_l > eval_result.semantic_similarity:
        return "WEAK_INFERENCE"  # 규정 재진술만 (추론 없음)

    if category == "multi_hop" and confidence < 0.5:
        return "MULTI_HOP_FAILURE"  # 관련 문서 못 찾음

    return "LOW_CONFIDENCE"  # 전반적 저조
```

**각 패턴별 자동 수정 제안:**

```python
FIX_SUGGESTIONS = {
    "WRONG_RETRIEVAL": [
        "소스 필터 키워드 추가",
        "top_k 증가 (20 → 25)",
        "인제스트 품질 확인 (청크 깨짐 여부)",
    ],
    "VERBOSE_ANSWER": [
        "max_tokens 감소 (2048 → 1024)",
        "간결성 few-shot 추가",
        "'200자 초과 금지' 프롬프트 강화",
    ],
    "HALLUCINATION": [
        "Self-RAG 활성화",
        "temperature 낮추기",
        "'컨텍스트에 없는 내용 생성 금지' 강화",
    ],
    "WRONG_SCOPE": [
        "부정 few-shot 추가",
        "'거짓 긍정 > 거짓 부정' 원칙 강화",
        "neg_markers 패턴 추가",
    ],
    "WEAK_INFERENCE": [
        "추론 few-shot 추가",
        "'이유를 설명하세요' 프롬프트 추가",

    ],
```

### 알고리즘 8: 반복 튜닝 루프 (Iterative Tuning Loop)

전체 프로세스를 자동화한 반복 루프입니다.

```python
async def iterative_tuning(
    golden_dataset,
    target_success_rate=0.95,
    max_iterations=3,
):
    """
    반복 튜닝 루프

    1. 벤치마크 실행
    2. 실패 문항 분석
    3. 패턴별 자동 수정 적용
    4. 다시 벤치마크
    5. 목표 도달 또는 최대 반복까지 계속
    """

    for iteration in range(max_iterations):
        # Step 1: 벤치마크
        results = await run_benchmark(golden_dataset)
        success_rate = results.success_count / results.total_count

        if success_rate >= target_success_rate:
            return f"목표 달성: {success_rate:.1%} (반복 {iteration+1}회)"

        # Step 2: 실패 분석
        failures = [r for r in results if not r.success]
        patterns = Counter(classify_failure(f) for f in failures)

        # Step 3: 우선순위별 수정
        # 가장 빈번한 패턴부터 해결
        for pattern, count in patterns.most_common():
            suggestions = FIX_SUGGESTIONS[pattern]
            apply_top_suggestion(suggestions[0])  # 가장 효과적인 수정 적용

        # Step 4: 변경 기록
        log_changes(iteration, patterns, applied_fixes)

    return f"최대 반복 도달: {success_rate:.1%}"
```

**주의 사항:**
- 각 반복에서 **하나의 변경만** 적용 (격리 원칙)
- 변경 후 성능이 **악화**되면 즉시 롤백
- 모든 변경은 로그에 기록 (재현성 확보)

### 알고리즘 9: Composite Score 계산 흐름도

전체 평가 파이프라인을 한눈에 보여주는 흐름도입니다.

```
[입력] 골든 답변 + 생성 답변
         │
         ├─→ bge-m3 임베딩 → 코사인 유사도 → semantic_similarity
         │
         ├─→ 한국어 글자 토크나이저 → ROUGE-1 F1 → rouge_1
         │                          → ROUGE-L F1 → rouge_l
         │
         └─→ 길이 비교 → length_penalty

[기본 Composite]
  = (0.65 × semantic + 0.20 × rouge_l + 0.15 × rouge_1) × length_penalty

[카테고리 조정]
  카테고리별 가중치 적용 → category_score

[부정 감지]
  category == "negative"?
    YES → neg_markers 매칭 → 거짓 긍정이면 × 0.3
    NO  → 그대로

[성공/실패 판정]
  category_score >= threshold?  (factual: 0.45, negative: 0.40, ...)
    AND confidence > conf_threshold?  (factual: 0.15, negative: 0.0, ...)

  YES → ✅ 성공
  NO  → ❌ 실패 → 실패 패턴 분류
```

## 10. 실패 패턴 10가지와 해법

실패 분석을 자동화했습니다. 모든 실패 문항을 10가지 패턴으로 분류하고, 각 패턴에 맞는 해법을 제안합니다.

### 패턴 분류표

| # | 패턴 | 감지 조건 | 해법 | 우선순위 |
|---|------|---------|------|---------|
| 1 | **WRONG_RETRIEVAL** (검색 실패) | confidence < 0.3 | 소스 필터 키워드 추가, top_k 증가 | 최고 |
| 2 | **VERBOSE_ANSWER** (답변 과잉) | length_penalty < 0.9 | max_tokens 감소, 간결성 few-shot | 높음 |
| 3 | **HALLUCINATION** (환각) | semantic < 0.4 | Self-RAG 활성화, temperature 낮춤 | 높음 |
| 4 | **WRONG_SCOPE** (범위 오류) | negative + semantic > 0.4인데 실패 | 부정 few-shot 추가, 범위 규칙 강화 | 높음 |
| 5 | **WEAK_INFERENCE** (추론 부족) | inference + rouge > semantic | 추론 few-shot, "이유를 설명" 지시 | 중간 |
| 6 | **MULTI_HOP_FAILURE** (다중연결 실패) | multi_hop + confidence < 0.5 | Multi-Query 활성화, top_k 30 | 중간 |
| 7 | **INCOMPLETE_LIST** (목록 누락) | 목록 마커 + 길이 70% 미만 | "모든 항목을 빠짐없이" 지시 | 중간 |
| 8 | **LANGUAGE_LEAK** (언어 유출) | 비한국어 문장 감지 | 금지 규칙 + 후처리 정규식 | 낮음 |
| 9 | **FORMAT_ERROR** (형식 오류) | "A:", "답변:" 접두사 존재 | 후처리 정규식 | 낮음 |
| 10 | **LOW_CONFIDENCE** (전반적 저조) | 명확한 패턴 없음 | 인제스트 품질, 청크 사이즈 점검 | 낮음 |

### 해결 우선순위

```
1순위: WRONG_RETRIEVAL → 검색이 안 되면 아무것도 안 됨
2순위: VERBOSE_ANSWER  → 가장 빈번한 실패
3순위: HALLUCINATION   → 가장 위험한 실패
4순위: WRONG_SCOPE     → negative 카테고리 전용
5순위: 나머지         → 도메인별 맞춤 대응
```

## 11. Advanced RAG는 정말 필요한가?

### 파라메트릭 서치 결과

21가지 설정 조합을 그리드 서치로 테스트했습니다.

#### Speed 테스트 (속도 vs 품질)

| 설정 | 성공률 | Composite | 평균 응답시간 |
|------|--------|-----------|-------------|
| baseline (top_k=20, rerank=5) | 85% | 0.613 | 175초 |
| light (top_k=10, rerank=3) | **85%** | **0.654** | **108초** |
| minimal (top_k=5, rerank=3) | 75% | 0.639 | **22초** |
| no_rerank | 최저 | 최저 | 가장 빠름 |

**발견:**
- `top_k=10, rerank=3`이 baseline보다 **성능도 높고 속도도 빠름**
- 리랭커를 끄면 성능이 확 떨어짐 → **리랭커는 필수**
- `top_k`을 너무 낮추면(5) 성공률 하락

#### Advanced RAG 기능별 효과

```
기본 RAG (Vector + BM25 + Reranker):     93.3% ✅
+ Multi-Query:                           효과 미미 (1-2% 차이)
+ Self-RAG:                              환각 감소 (특정 문항만)
+ HyDE:                                  효과 불분명
+ Agentic RAG:                           오버헤드 > 개선 효과
+ 전부 활성화:                           속도 2x 느려짐, 개선 1-2%
```

### 결론

> **93.3%는 "기본 RAG + 좋은 프롬프트 + 좋은 평가"만으로 달성했습니다.**
> Advanced RAG 기법들은 마지막 3-4%를 위한 것이지, 기본기를 대체하지 않습니다.

투자 대비 효과:

```
프롬프트 엔지니어링:  60% → 100% (투자 대비 효과 최고)
도메인 Few-Shot:      93% → 100% (투자 대비 효과 높음)
소스 필터링:          교차 오염 문제 해결 (필수)
길이 패널티:          장황한 답변 감점 (필수)
Advanced RAG:         93% → 94% (투자 대비 효과 낮음)
```

## 12. 자동화 파이프라인

수동 튜닝의 한계를 느끼고, **8단계 자동화 파이프라인**을 구축했습니다.

### 파이프라인 구조

```
Step 1: INGEST            → 문서 인제스트
Step 2: GENERATE_GOLDEN   → LLM 기반 골든 데이터셋 자동 생성
Step 3: VALIDATE_DATASET  → 카테고리 균형, 중복, 필수 필드 검증
Step 4: BASELINE_BENCHMARK → 현재 성능 측정
Step 5: ANALYZE_FAILURES   → 10가지 실패 패턴 자동 분류
Step 6: AUTO_TUNE          → 실패 분석 기반 파라미터 자동 조정
Step 7: TUNED_BENCHMARK    → 튜닝 후 성능 측정
Step 8: FINAL_REPORT       → HTML/JSON 결과 리포트 생성
```

### 반복 튜닝

```python
target_success_rate = 0.95
max_iterations = 3

while current_rate < target_rate and iteration < max_iterations:
    failures = analyze_failures(results)
    apply_suggestions(failures)
    current_rate = run_benchmark()
    iteration += 1
```

목표 성공률(95%)에 도달하거나 3회 반복까지 자동으로 튜닝합니다.

### 실패 분석기 자동 제안 예시

```
[VERBOSE_ANSWER 3건 감지]
제안:
  1. max_tokens를 2048 → 1024로 줄이기
  2. 간결성 few-shot 2개 추가
  3. 시스템 프롬프트에 "200자 초과 금지" 강화

[WRONG_RETRIEVAL 2건 감지]
제안:
  1. 해당 도메인의 auto-detect 키워드 추가
  2. top_k를 20 → 25로 증가
  3. 인제스트 품질 확인 (청크 깨짐 여부)
```

## 13. 핵심 교훈 요약

### 가장 중요한 것 Top 5

| 순위 | 교훈 | 근거 |
