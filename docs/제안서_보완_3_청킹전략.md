# (3) 4가지 청킹 전략 상세

## 위치: II. 제안시스템 구축 > 2.2 핵심 기술 > 청킹 전략

현재 제안서에는 4종 전략이 표로 간략히 소개되어 있음 (line 147~157). 아래는 각 전략을 **상세 기술**한 보완 내용.

---

## 개요

문서를 벡터 DB에 저장하려면 적절한 크기의 청크(chunk)로 분할해야 하며, 청크의 크기와 분할 방식은 검색 정확도와 답변 품질에 직접적인 영향을 미친다.

- **청크가 너무 크면**: 불필요한 정보가 포함되어 LLM이 핵심 내용을 추출하지 못하고 답변 정확도 하락
- **청크가 너무 작으면**: 맥락이 유실되어 단편적이고 불완전한 답변 생성

이에 본 플랫폼은 **문서 유형별 특성**을 고려하여 4종 청킹 전략을 차등 적용한다.

---

## 전략 1: Semantic Chunking (의미 기반 분할)

### 적용 대상
- **사규/지침 (HWP)**: 절차 단계별 의미 경계가 명확한 문서
- 예: 안전관리규정, 검사절차서, 업무매뉴얼

### 동작 원리
1. 문서를 문장 단위로 분리
2. 인접 문장 간 **임베딩 유사도**를 계산 (BGE-M3)
3. 유사도가 **급격히 하락하는 지점**을 의미 경계로 판정
4. 의미 경계를 기준으로 300~500 토큰 단위의 청크 생성
5. 인접 청크 간 **50토큰 오버랩**으로 경계 부분 맥락 보존

### 파라미터
| 항목 | 값 | 설명 |
|------|-----|------|
| 청크 크기 | 300~500 토큰 | 의미 완결성 기준 동적 조절 |
| 오버랩 | 50 토큰 | 경계 맥락 유실 방지 |
| 유사도 임계값 | 코사인 유사도 기반 | 문서별 자동 조정 |
| 최소 청크 크기 | 100 토큰 | 지나치게 짧은 청크 방지 |

### 도메인 특화 처리
- 가스 설비·배관·용접 등 **공사 고유 기술 용어 사전**을 청킹 과정에 적용
- 도메인 특화 용어가 포함된 문장이 의미적으로 분리되지 않도록 처리
- 예: "배관 용접부 비파괴검사(RT, UT, MT, PT)"가 중간에서 끊기지 않도록 보호

### 장점
- 절차서의 단계별 의미 단위가 보존됨
- 검색 시 관련 절차가 하나의 청크로 반환되어 정확한 답변 가능
- 기계적 분할 대비 답변 품질 향상

## 전략 2: Fixed-size Chunking (고정 크기 분할)

### 적용 대상
- **기술 매뉴얼 (DOCX/PDF)**: 긴 연속 서술형 문서
- **보고서**: 국외출장 결과보고서, 기술 검토 보고서
- 예: ISO 표준 문서, 설비 운영 매뉴얼

### 동작 원리
1. 문서를 **문단(paragraph) 단위**로 1차 분리
2. 각 문단을 **512 토큰** 고정 크기로 분할
3. 인접 청크 간 **100토큰 오버랩**으로 문맥 연결 보존
4. 문단 경계에서는 우선적으로 분할하여 의미 단절 최소화

### 파라미터
| 항목 | 값 | 설명 |
|------|-----|------|
| 청크 크기 | 512 토큰 | 고정 크기 |
| 오버랩 | 100 토큰 | 긴 서술문의 문맥 보존 |
| 분할 우선순위 | 문단 > 문장 > 문자 | 계층적 분할 |
| 최대 청크 크기 | 600 토큰 | 오버랩 포함 상한 |

### 계층적 분할 순서
```
1순위: 문단 경계 (\n\n)에서 분할
2순위: 문장 경계 (. ! ? 등)에서 분할
3순위: 문자 단위 분할 (최후 수단)
```

### 장점
- 구현이 단순하고 예측 가능한 청크 크기
- 긴 서술형 문서에서 균일한 검색 품질 보장
- 대량 문서 배치 인제스트 시 안정적 성능

## 전략 3: Table Chunking (표 전용 분할)

### 적용 대상
- **표/테이블이 포함된 모든 문서**: ALIO 공시 데이터, 배관 스펙 시트, 예산 결산표
- 예: 자본금 현황표, 인력 현황표, 설비 점검 이력표, 예산 집행 내역

### 동작 원리
1. Upstage Document Parse가 문서 내 **표 요소(category: table)를 자동 식별**
2. 표 전체를 **하나의 청크**로 보존 (행/열 분리 방지)
3. 표 앞뒤의 **캡션/제목**을 청크 메타데이터에 포함
4. 대형 표(1000토큰 초과)는 **행 그룹 단위**로 분할하되 헤더 행 복제

### 파라미터
| 항목 | 값 | 설명 |
|------|-----|------|
| 기본 단위 | 표 전체 | 구조 보존 우선 |
| 최대 청크 크기 | 1000 토큰 | 초과 시 행 그룹 분할 |
| 헤더 복제 | O | 분할된 각 청크에 헤더 행 포함 |
| 캡션 포함 | O | 표 제목/설명을 메타데이터에 첨부 |

### 대형 표 분할 예시
```
원본: 100행 × 5열 테이블 (2000토큰)
  ↓ 분할
청크 1: 헤더(1행) + 1~30행 = ~700토큰
청크 2: 헤더(1행) + 31~60행 = ~700토큰
청크 3: 헤더(1행) + 61~100행 = ~700토큰
  → 각 청크에 헤더가 복제되어 독립적으로 해석 가능
```

### 도메인 특화 처리
- ALIO 공시 데이터의 재무제표 → 표 전체를 하나의 청크로 보존하여 "자본금은 얼마인가?" 같은 팩트 질의에 정확한 수치 답변
- 배관 스펙 시트의 다단 표 → Upstage가 병합 셀/숨겨진 격자선을 복원한 후 테이블 청킹 적용

### 장점
- 표의 행/열 관계가 보존되어 수치 팩트 질의에 정확한 답변
- 표 구조 파괴로 인한 환각(잘못된 수치 인용) 방지
- 검색 시 표 캡션을 통한 관련도 향상

## 전략 4: Hierarchical Chunking (부모-자식 청킹)

### 적용 대상
- **법령/사규**: 조·항·호 계층 구조를 가진 법률 문서
- 예: 한국가스공사법, 도시가스사업법, 고압가스안전관리법, 내부 사규/지침

### 동작 원리
1. 법령/사규의 **조문 체계**(제○조 > 제○항 > 제○호)를 파싱
2. **자식 청크**: 항·호 단위의 세밀한 청크 (200토큰) — **검색용**
3. **부모 청크**: 조 단위의 상위 청크 (1000토큰) — **LLM 컨텍스트용**
4. 검색 시 자식 청크로 정확한 조문을 찾되, LLM에는 부모 청크를 전달

### 파라미터
| 항목 | 값 | 설명 |
|------|-----|------|
| 자식 청크 크기 | 200 토큰 | 항·호 단위 정밀 검색 |
| 부모 청크 크기 | 1000 토큰 | 조 단위 맥락 보존 |
| 매핑 | 자식 → 부모 링크 | 검색 결과에서 부모 추적 |
| 메타데이터 | 조·항·호 번호 | 정확한 출처 표시 |

### 동작 예시
```
문서: 한국가스공사법

부모 청크 (제15조 전체, ~800토큰):
  "제15조(천연가스의 수입) ①공사는... ②제1항에 따라... ③산업통상자원부장관은..."

자식 청크 1 (제15조 제1항, ~150토큰):
  "①공사는 천연가스를 직접 수입하거나..."

자식 청크 2 (제15조 제2항, ~120토큰):
  "②제1항에 따라 천연가스를 수입하려는..."

자식 청크 3 (제15조 제3항, ~180토큰):
  "③산업통상자원부장관은 제2항에 따른..."

[검색 흐름]
질의: "천연가스 수입 절차는?"
  → 자식 청크 2 (제15조 제2항)가 최고 유사도로 검색됨
  → LLM에는 부모 청크 (제15조 전체)를 전달
  → LLM이 제1항 배경 + 제2항 절차 + 제3항 예외를 종합하여 정확한 답변 생성
```

### 장점
- 조문 단위 정밀 검색과 조·항·호 간 맥락 보존을 **동시 충족**
- 조문의 단편적 인용으로 인한 **오해석 위험 방지**
- 법률 질의에서 "제○조 제○항"으로 정확한 출처 표시 가능

## 4가지 전략 종합 비교표

제안서 [표] 대체용:

| 전략 | 적용 대상 | 청크 크기 | 오버랩 | 핵심 효과 |
|------|----------|----------|--------|----------|
| **Semantic** | 사규/지침(HWP), 절차서 | 300~500토큰 (동적) | 50토큰 | 절차 단계별 의미 단위 보존, 도메인 용어 보호 |
| **Fixed-size** | 기술 매뉴얼, 보고서 | 512토큰 (고정) | 100토큰 | 문단→문장→문자 계층적 분할, 균일 품질 |
| **Table** | 표/테이블 포함 문서 | 표 전체 (최대 1000토큰) | - | 행/열 관계 보존, 수치 팩트 정확도 |
| **Hierarchical** | 법령/사규 (조·항·호) | 자식 200 / 부모 1000토큰 | - | 정밀 검색 + 맥락 보존 동시 충족 |

## 문서 유형별 전략 자동 선택

| 문서 유형 | source_type | 1순위 전략 | 2순위 전략 | 비고 |
|----------|-------------|-----------|-----------|------|
| 법령 | law | Hierarchical | - | 조·항·호 파싱 |
| 사규/지침 (HWP) | regulation | Semantic | Hierarchical | 조문 체계 있으면 Hierarchical |
| 기술 매뉴얼 | manual | Fixed-size | Semantic | 긴 서술형 |
| 보고서 | report | Fixed-size | Semantic | 국외출장 등 |
| ALIO 공시 | alio | Table | Fixed-size | 재무제표/통계표 중심 |
| 인쇄홍보물 | brochure | Semantic | Fixed-size | 마케팅/홍보 문구 |
| ISO 표준 | standard | Fixed-size | Table | 표 포함 시 Table 병행 |

> **자동 선택 로직**: 문서 업로드 시 메타데이터의 `source_type`에 따라 청킹 전략이 자동 결정됨. 하나의 문서 내에서 본문은 Semantic/Fixed-size, 표 영역은 Table이 **병행 적용**될 수 있음.
