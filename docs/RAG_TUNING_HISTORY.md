## Phase 1: 한국가스공사법
**기간**: 2026-02 초반
**데이터**: 한국가스공사법 PDF + 관련법 요약 (28 청크)
**모델**: Ollama qwen2.5:14b (로컬)
**결과**: 50문항 → **100% (50/50)**

### 주요 튜닝
| 항목 | 변경 전 | 변경 후 | 효과 |
|------|---------|---------|------|
| 시스템 프롬프트 | 일반 한국어 답변 | 법률 전문 프롬프트 (`rag_legal_system`) | 조문 인용 정확도 ↑ |
| Few-shot 예시 | 0개 | 법률 도메인 6개 | 답변 형식 일관성 ↑ |
| 모델 크기 인식 | 없음 | 7B 모델용 추가 간결성 지시 | 토큰 낭비 ↓ |
| 카테고리별 가중치 | 균등 | factual/inference/negative 별도 가중치 | 평가 정밀도 ↑ |
| 길이 패널티 | 없음 | 2.5x 초과 답변 감점 | 장황한 답변 ↓ |

### 핵심 발견
- **조문 원문 인용** 지시가 가장 큰 효과 — "한 글자도 바꾸지 말고 그대로 복사"
- 유사어 치환(편익→편의) 할루시네이션 빈번 → 명시적 금지 규칙 추가
- 법률명 매칭 규칙 추가 (질문의 법률명 ↔ 파일명 매칭)

## Phase 2: 내부규정 60문항
**기간**: 2026-02 중반
**데이터**: 676 HWP 내부규정 (19,035 청크)
**모델**: Ollama qwen2.5:14b (로컬)
**결과**: 60문항 → **98.3% (59/60)**

### 주요 튜닝
| 항목 | 변경 전 | 변경 후 | 효과 |
|------|---------|---------|------|
| 다중 규정 연계 | 미지원 | "두 개 이상 규정 연결" 프롬프트 추가 | multi_hop 정확도 ↑ |
| 추론 거부 방지 | "확인되지 않습니다" | "추론 질문에는 절대 확인되지 않습니다로 답하지 마세요" | inference 정확도 ↑ |
| Few-shot 확대 | 6개 | 8개 (factual + negative + inference + multi_hop) | 전 카테고리 커버 |
| 산술 계산 허용 | 미허용 | "변동액/차이/증감 계산" 명시 | 수치 비교 질문 정확도 ↑ |
| 응답 검증 | 없음 | 깨진 출력 감지 + 자동 재시도 | 불완전 응답 0% |

### 부정 질문 처리 체계 도입
- "~에 관한 규정이 있나요?" 형태 질문 전용 처리 로직
- "단순 단어 언급 ≠ 전담 규정 존재" 규칙 확립
- **"거짓 긍정보다 거짓 부정이 낫습니다"** 원칙 도입 (→ Phase 5에서 부작용 발견)

### 핵심 발견
- 내부규정은 법률과 달리 **비구조화** → 다중 규정 연계 처리 필수
- 파일명에 규정명이 포함 → 파일명 매칭 규칙이 효과적
- 1건 실패: Q60 (복합 산술 + 다규정 연계) — 모델 한계

### 주요 튜닝
| 항목 | 변경 전 | 변경 후 | 효과 |
|------|---------|---------|------|
| source_type 필터 | 없음 | 질문 키워드 → 메타데이터 필터 자동 매칭 | 검색 정밀도 ↑ |

## Phase 4: RunPod 배포
**기간**: 2026-03 초
**데이터**: 동일 39,739 청크 (Milvus Lite)
**모델**: vLLM Qwen2.5-32B-Instruct-AWQ (A40 GPU)
**결과**: 120문항 → 28.3% → **80.0% (96/120)**

### 초기 문제점 (28.3%)
- Safety fallback 메시지가 답변 본문에 삽입 → 정확도 저하
- `confidence_low` 임계값 0.5 너무 높음 → 과도한 safety fallback
- Chinese text validation 30% 임계값 → Qwen 모델 중국어 소량 누출도 차단

### 단계별 개선
| 단계 | 변경 | 정확도 |
|------|------|--------|
| 초기 | RunPod 배포 그대로 | 28.3% |
| #1 | Safety fallback → 별도 SSE 이벤트 분리 | ~45% |
| #2 | confidence_low 0.5→0.3 | ~60% |
| #3 | Chinese validation 30%→5% | ~65% |
| #4 | .env 최적화 (CONTEXT_MAX_CHUNKS=5, MAX_TOKENS=512) | ~75% |
| #5 | FEW_SHOT_MAX_EXAMPLES=2, RERANK_TOP_N=5 | 80.0% |

### .env 최적화 (RunPod)
```bash
CONTEXT_MAX_CHUNKS=5      # 로컬 10 → 5 (불필요한 노이즈 감소)
LLM_MAX_TOKENS=512        # 로컬 1024 → 512 (간결한 답변 유도)
RERANK_TOP_N=5             # 재정렬 후 상위 5개만
FEW_SHOT_MAX_EXAMPLES=2    # 예시 2개 (토큰 절감)
```

### 잔여 실패 분석 (24/120)
| 유형 | 건수 | 설명 |
|------|------|------|
| wrong_no_regulation | 7 | "해당 규정이 없습니다" false negative |
| miss | 14 | multi_hop 매칭 실패 |
| false_positive | 1 | 없는 내용 생성 |
| 기타 | 2 | 모델 토큰 한계 |

## Phase 5: RAG평가용 문서 전용
**기간**: 2026-03-07
**데이터**: RAG평가용 문서 92파일 (84 HWP + 6 PDF + 2 서브디렉토리 HWP) → 10,518 청크
**모델**: vLLM Qwen2.5-32B-Instruct-AWQ (A40 GPU)
**결과**: 60문항 → 70.0% (42/60) → **재테스트 진행 중**

### 데이터 전환 결정
- 기존 5개 데이터셋(39,739 청크) 전량 폐기
- RAG평가용 문서 목록(92파일)만 단독 사용
- 이유: 평가위원이 사용할 실제 평가 문서에 최적화

### 인제스트
- HWP 파일: `dp_mode="force_dp"` (Upstage Document Parse)
- PDF 파일: `dp_mode="auto"` (로컬 PyPDF + 필요시 Upstage DP)
- 결과: 92/92 성공, 10,518 청크, 35분 소요

### 골든 데이터셋 (60문항)
- 51개 고유 규정 참조
- 카테고리: factual 25, inference 15, multi_hop 12, negative 8
- 난이도: easy 20, medium 25, hard 15
- 파일: `backend/tests/golden_dataset_rag_eval.json`

### Run 1 결과 (70.0%, 42/60)
| 카테고리 | 성공률 | 문제점 |
|---------|--------|--------|
| factual | 68% (17/25) | FALSE_NEG 다수 |
| inference | 53% (8/15) | FALSE_NEG + 추론 거부 |
| multi_hop | 83% (10/12) | 상대적으로 양호 |
| negative | 88% (7/8) | 양호 |

### FALSE_NEG 원인 분석
17건 실패 중 대부분이 **"이 법에 해당 규정은 없습니다"** 또는 **"확인되지 않습니다"** 형태.

**근본 원인**: 시스템 프롬프트의 부정 질문 처리 규칙이 너무 공격적
```
# 문제 규칙 (system.yaml)
"5. 확신이 없으면 반드시 \"확인되지 않습니다\"로 답하세요. 거짓 긍정보다 거짓 부정이 낫습니다."
```

이 규칙이 factual/inference 질문에도 적용되어, 컨텍스트에 답이 있는데도 "확인되지 않습니다"로 답하는 현상 발생.

### Run 2 프롬프트 수정 v1 (43/60, 71.7%)

- 효과: +1문항 (Q54 HTTP 500 → OK)

### 핵심 교훈 (Phase 5)
1. **골든 데이터셋 품질이 가장 중요** — 질문의 답이 실제 인제스트된 청크에 있는지 검증 필수
2. **프롬프트 수정만으로는 한계** — Run 2~3에서 +5문항 개선이 한계 (70%→75%)
3. **검색 실패는 질문 교체가 효과적** — top_k 증가보다 질문을 검색 가능한 형태로 변경이 실용적
4. **API 기반 대안 질문 테스트** — 자동으로 대안 질문 다수 테스트 → 통과하는 질문으로 교체
5. **단계별 접근이 핵심** — 프롬프트 → 골든 데이터셋 v1 → 골든 데이터셋 v2 순서로 점진 개선

## 핵심 교훈 요약

### 1. 프롬프트 엔지니어링
- **조문 원문 인용** 지시가 가장 효과적 — 의역/유사어 치환 금지
- **카테고리별 처리** 분화 필수 (factual ≠ inference ≠ negative)
- **"거짓 부정 > 거짓 긍정"** 원칙은 부정 질문에만 적용해야 함 — 일반 질문에 적용하면 역효과
- 모순되는 규칙 존재 시 LLM은 보수적 규칙(거부)을 선호 → 우선순위 명시 필요
- 법률명 ↔ 파일명 매칭 규칙이 검색 정밀도에 큰 영향

### 2. 검색(Retrieval) 튜닝
- **Hybrid 검색** (Vector + BM25)이 단일 vector보다 우수
- **동적 가중치 융합**: 쿼리 유형별 vector/BM25 비율 자동 조정
- **source_type 필터**: 질문 키워드 → 메타데이터 필터 자동 매칭으로 노이즈 감소
- CONTEXT_MAX_CHUNKS: 너무 많으면 노이즈 ↑, 너무 적으면 recall ↓ → 5가 최적

### 3. 모델/인프라
- **Reranker 역효과 가능**: USE_RERANK=false가 오히려 나은 경우 존재
- **vLLM 캐시 워밍업**: 첫 쿼리 후 속도 3배 개선 (10.3s → 3.5s)
- **Chinese text 누출**: Qwen 모델 특성 — validation 임계값 5%로 차단
- **Safety fallback**: 답변 본문 삽입 ↓ → 별도 SSE 이벤트로 분리

### 4. 데이터
- HWP 파일은 반드시 **Upstage Document Parse** 사용 (로컬 파서 불완전)
- `dp_mode="force_dp"` for HWP, `dp_mode="auto"` for PDF
- Milvus Lite는 single-process lock → 인제스트 시 서버 정지 필요
- ChromaDB → Milvus 전환 시 디스크 51% 절감

### 5. 평가 체계
- **골든 데이터셋** 기반 자동 벤치마크가 튜닝 효과 측정에 필수
- 카테고리별 분석 (factual/inference/multi_hop/negative) 없이는 문제 진단 불가
- FALSE_NEG 자동 감지 (특정 문구 패턴 매칭)가 효과적
- 실패 사례 분석 → 프롬프트 수정 → 재벤치마크 사이클 반복
